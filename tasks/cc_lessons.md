# CC Lessons

- When a user corrects a geometric edge case, prefer extending the core math with a deterministic fallback and add regression tests for that edge case immediately. Example: parallel infinite bars should not raise in `geometry.py`; use the Appendix A parallel-normal construction for distance and return a canonical closest-point parameter pair instead of failing.
- When a user marks a tool or script as obsolete, remove the implementation artifact and prune the user-facing README workflow/docs references in the same pass so install instructions do not keep advertising dead paths.
- When documenting Rhino 8 CPython setup, do not tell users to trust `sys.executable` from ScriptEditor. It can report `Rhino.exe`; document the deployed RhinoCode runtime under `.rhinocode/py39-rh8` instead, and mention `RhinoCodePlugin.RootPath` for custom installs.
- When documenting a Rhino workflow that depends on preauthored frame annotations and block definitions, point users to the exact starter `.3dm` file and explain which artifacts it contributes. Also say explicitly when `export_cad_config.py` must be rerun, especially after joint-definition changes or when adapting to a different joint family.
- When a user asks for FK visualization, do not draw independently solved endpoint frames if the intended behavior is a kinematic tree. Build the viewer as a true parent-child chain so upstream joint motion propagates to all downstream links, URDF-style.
- When a user asks for comparison geometry in a viewer, avoid stacking the ghost copy directly on top of the source model. Offset the comparison copy spatially and send numeric diagnostics to the terminal instead of blinking in-scene text overlays.
- When a Rhino command uses core geometry helpers that already support an edge case, remove redundant script-level rejection logic and let the shared geometry path handle it. Example: `t1_bar_axis.py` should not reject parallel bars once `closest_params_infinite_lines` supports them.
- When a user specifies the exact package and version for a new dependency, add that pinned dependency to the repo manifest first and install from the manifest rather than doing an ad-hoc package install. Example: use `pybullet_planning==0.6.1` in `requirements.txt` for the PyBullet viewer workflow.
- When an all-solutions geometry solver visualizes overlapping branches, do not assume the sign pairs uniquely identify geometry. Keep all distinct converged minima across seeds, dedupe by geometric equivalence, and add a regression for any missing mirrored branch.
- When a pytest visualization mode pauses on `input()` or opens interactive viewers, make `--viz` disable capture automatically so `pytest ... --viz` behaves like `-s` without requiring a second flag.
- Rhino 8 Python 3 scripts must be launched through the ScriptEditor run command in toolbar macros and docs. Do not wire them to `_-RunPythonScript`, which uses the legacy parser and will fail on Python 3 syntax like function annotations.
- Rhino CPython interop should not assume Rhino point values are plain tuples. Convert `Point3d`-like objects explicitly via `.X/.Y/.Z` before passing them into NumPy, or the scripts will fail on selections and picked points.
- When a Rhino workflow starts from a user-selected bar that should be repositioned, preserve its original length and attributes by moving the existing object instead of deleting it and recreating a default-length replacement.
- When users want construction geometry preserved in Rhino, bake explicit reference objects at the moment the result is finalized. For shortest-contact segments, assign a distinct object color and set the object print width source to by-object so the weight is not inherited from the layer.
- When Rhino shows multiple candidate solutions for user selection, preview the supporting construction geometry for each candidate too, and match its color to the candidate axis so the grouping stays visually unambiguous.
- When Rhino placeholder joint blocks are centered on the kinematic OCF, they can visually overlap even when the FK is correct. Keep the stored OCF transform unchanged, but apply a placeholder-only local visual correction so each box occupies its own side of the interface and the bar-side face aligns with the supporting bar axis.
- When a revolute DOF like `JJR` needs to be visually readable in fallback placeholder blocks, do not use symmetric square footprints. Give the placeholder different X/Y extents so the rotation shows up as a clear edge alignment change.
- When Rhino FK visuals are still ambiguous, bake the exact solver frames as RGB axis tripods directly from the matrices used in code. That separates frame-convention bugs from placeholder-geometry bugs quickly.
- When a PyBullet viewer needs link-frame debugging, prefer `pybullet_planning` helpers over raw debug primitives: load and pose the URDF through `pp`, use `pp.get_link_pose` plus `pp.draw_pose` for frames, `pp.add_text`/`pp.get_link_name` for labels, and `pp.set_color`/`pp.apply_alpha` for temporary visualization styling.
- When PyBullet link/frame annotations are only for inspection, draw them once in a small static viewer script instead of redrawing them every frame in the interactive viewer loop. Repeated `draw_pose`/debug text updates can make the main viewer sluggish for no benefit.
- When labeling PyBullet links in a static viewer, place each name from the link's world pose instead of only attaching a fixed local text offset to the link. Invisible carrier links often share or nearly share transforms, so stagger duplicate-pose labels slightly to keep them readable.
- When a PyBullet static viewer is meant to inspect one connector segment, do not annotate the whole URDF by default. Filter to an explicit link-name span so the visualization stays legible and focused.
- When narrowing a static URDF viewer to a connector segment, include any requested reference bars like `le_bar` and `ln_bar` explicitly in the annotation list so the local connector frames still have context.
- When a Rhino helper gains a user-facing naming parameter, make it optional with an empty default and only bake the label geometry when the user actually enters a name. Example: frame-baking scripts should prompt for a frame name, default to empty, and add a text dot at the origin only when that name is non-empty.
- When a Rhino frame label should read relative to the constructed frame, place the text dot using the frame axes instead of world coordinates. Example: offset the frame name one axis length along local `-X` rather than baking it directly at the origin.
- When a Rhino construction helper bakes multiple related objects as one result, group them immediately after creation so selection and downstream transforms treat them as a single unit. Example: frame axis lines and the optional frame-name text dot should be added to one fresh Rhino group.
- When a Rhino label offset needs independent control, expose it as its own millimeter parameter instead of coupling it to another size setting. Example: a frame-name text dot should use a dedicated offset prompt with its own default rather than reusing axis length.
- When users provide baked CAD meshes that are already aligned to connector link frames, use those mesh assets directly for URDF and static-viewer visuals instead of placeholder boxes, but keep kinematic parameter extraction driven by the annotated frames rather than by parsing the mesh geometry.
- When a user corrects URDF joint conventions, treat the explicit parent/child order, joint axis, and axis-remap description as the source of truth for the chain design. Mirror those exact transforms in both FK and URDF, and call out any impossible axis remap immediately instead of silently inferring a symmetric replacement.
- When a user confirms a frame remap but leaves one axis unspecified, complete the remaining axis by enforcing a right-handed orthonormal basis and record the resolved remap explicitly in config and tests. Example: if `old x -> new z` and `old y -> new x`, then `old z -> new y`.
- When a user clarifies that a fixed joint includes a real rotation as well as translation, stop validating it as translation-only. Encode the full measured transform, add a regression for the intended rotation pattern, and let FK/URDF consume the same transform directly.
- When generated CAD parameters and hand-maintained limits live side by side, keep one stable public config module for runtime imports and one generated data file for exporter output, and document clearly which file is authoritative for geometry versus manual settings so users do not think the wrapper module is obsolete.
- When exporting URDF transforms from floating-point CAD math, clamp tiny origin `xyz` and `rpy` components to exactly `0.0` at serialization time instead of letting `1e-16`-scale noise leak into the XML. Keep that clamp scoped to URDF origins so unrelated numeric fields like mesh scale are not silently altered.
- When a user corrects the unit convention of baked CAD meshes, keep the fix local to URDF export: point the mesh visuals at the `*_mm.obj` assets and apply an explicit `0.001` mesh scale there instead of assuming the generated config mesh paths already match the viewer/export units.
- When the URDF chain is directional but the solver concept is "two sides meeting at the interface," make the FK API bar-frame driven from each side toward the center and constrain only the interface quantities that are physically meaningful. For coaxial screw-hole interfaces, optimize on origin alignment plus shared local `Z`, and do not overconstrain the free twist about that axis.
- When an interactive viewer runs checks continuously, do not leave the user guessing whether anything happened. Print a small help banner at startup, add explicit hotkeys to dump detailed reports, and surface optimizer convergence summaries when the solve refreshes so the live status line is not the only feedback channel.
- When a user refines Rhino construction output organization, use the exact shared category layers they ask for instead of inventing mode-specific ones, and pipe every involved bar axis with flat caps so the tube starts and ends exactly at the axis line endpoints.
- When Rhino placement should use real CAD link blocks instead of placeholders, require the exact block definition names the user provided and insert those existing definitions directly at the solved link frames. Do not silently recreate fallback box blocks once the document is expected to own the visual assets.
- When Rhino bake helpers assign objects to named output layers, centralize the layer creation check inside the shared display-assignment helper so every object type, including points and contact segments, works even in a blank document.
- When a Rhino tube must match a straight bar-axis line exactly, do not rely on a generic pipe helper alone. Build the tube from the line endpoints as a capped cylinder/Brep so the solid starts and ends exactly at the axis segment bounds.
- When Rhino commands depend on tunable CAD config constants, do not import scalar values like `BAR_CONTACT_DISTANCE` by value at module import time. Reload the config module at command execution and read attributes from it so reruns in the same Rhino session pick up the latest exported numbers.
- When a Rhino command offers a multi-solution preview and the user wants full solids visible, keep those preview solids, but suspend redraw while generating and cleaning them up. For `t1_bar_axis.py`, build the preview tubes with `EnableRedraw(False)`, redraw once after the batch is ready, and only re-enable viewport updates during the actual pick step.
- When a Rhino ScriptEditor command depends on local Python modules whose return schema changed, do not trust Rhino's cached imports across reruns. Reload the local modules at command start and keep a small compatibility fallback for newly added result fields so one stale import does not crash the command with a `KeyError`.
- When a user wants Rhino construction outputs organized for downstream editing, put placed instances and baked debug geometry on the exact named layers they asked for, create those layers on demand, and group each baked frame's axis lines plus text dot so every frame stays selectable as one unit.
- When a Rhino command wraps a multi-start optimizer, print the seed count, best objective, iteration/evaluation counts, and solver termination message in the Rhino command line after the solve so users can tell how the run converged without opening a separate viewer.
- When a Rhino placement command has several discrete post-solve assembly variants at the same contact location, present them as temporary numbered preview candidates with redraw suspended, let the user click one, then delete the previews and bake only the chosen final blocks plus frame annotations.
- When a connector branch is generated by flipping the male link 180 degrees about `male_link` local `Z`, do not treat it as a pure rigid post-solve variant. Keep `mjp/mjr` fixed, recompute the target `male_screw_hole_link` frame from the flipped male link, and reoptimize the female side against that target; only the paired female 180 degree twist can be derived as a free post-solve variant.
- When an optimization variable is only an indirect proxy for the geometry you actually constrain, switch to the direct geometric parameters. For `solve_s2_t1_all`, parameterize the two contact points directly by circle angles on the `Ce1/Ce2` contact circles, optimize the two orthogonality residuals separately, and derive the four branches by `+pi` flips of the converged angle pair.
- When a Rhino-side solver still misses branches in some interactive cases, add a one-click debug export that captures the exact selected geometry, config values, and current solver output as JSON. For `T1-S2`, export `Le1`, `Le2`, `Ce1`, `Ce2`, `BAR_CONTACT_DISTANCE`, and the live `solve_s2_t1_all(...)` results so the failing case can be replayed outside Rhino.
- When a replayed `T1-S2` debug case returns fewer than four solutions, do not assume the optimizer missed branches. First confirm the actual branch count with a dense seed sweep or objective scan, because some skew pairs of equal-radius contact circles only admit two real common tangents. In those cases the fix is better diagnostics and branch-family reporting, not forcing four outputs.
- When a Rhino user needs insight into why `T1-S2` returned only some branches, print the full solver diagnostics to the Rhino command line for every run. Include each solution's sign family, total objective, per-contact objective components, axial offsets, and angle-to-orthogonality errors, and explicitly list any missing sign families when fewer than four solutions exist.
- When listing missing `T1-S2` sign families, do not stop at the family names. Also run and print the corresponding missing-family optimization attempts so Rhino shows where those seeds actually converged, with the same `total_obj`, per-contact objective, offset, and orthogonality-angle diagnostics as the successful families.
- When optimization diagnostics are shown to a Rhino user, include the actual solver stopping criteria and stopping reason, not just the final objective. For `T1-S2`, print the active `xtol`, `ftol`, `gtol`, and `max_nfev`, plus each attempt's `nfev`, status, and SciPy termination message so a local minimum is not mistaken for a max-iteration failure.
- When the user wants `T1-S2` to converge only on gradient tolerance, disable the other least-squares stopping tests instead of merely tightening them. For SciPy `least_squares` with non-`lm` methods, set `ftol=None` and `xtol=None`, keep `gtol` finite, and rely on `max_nfev` as the remaining hard stop.
